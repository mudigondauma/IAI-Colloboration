01End-to-end data flow
Data moves through six zones — from raw source systems on the left to the two cockpit presentation views on the right. Every entity in this document maps to one of these zones. The unified data model means both views read from the same underlying tables; the difference is which fields and aggregations each view surfaces.

04Phase 1 alignment extensions
The current Phase 1 cockpit introduced four new shared needs that were previously implicit: a measurement spine, a finance-grade value spine, a persistent segmentation / action spine, and an explicit viewer-access spine. The new guided Ask-the-cockpit layer and role-specific saved views now ride on those same spines, so the model stays grounded without adding a separate chat-only or preset-only entity.

Design decisions · Every entity carries org_id for multi-tenancy. · Every time-series entity has period_date for trend tracking. · The Strategic and Delivery views read from the same underlying tables — the toggle changes which fields and aggregations are surfaced. · The guided Ask layer intentionally reuses METRIC_DEFINITION, METRIC_OBSERVATION, EVIDENCE_PACK, EXECUTIVE_ACTION, SEGMENT_SLICE, and ACCESS_SCOPE rather than creating a separate chat-only store. · The saved-view layer intentionally reuses SEGMENT_SLICE, ACCESS_SCOPE, and PORTFOLIO rather than creating a separate preset entity in the current prototype. · SCORE_NARRATIVE uses a polymorphic pattern so any scoreable entity can have plain-language explanations without separate tables. · LLM_OPERATION_LOG is append-only and should be partitioned by executed_at month in PostgreSQL for cost.
